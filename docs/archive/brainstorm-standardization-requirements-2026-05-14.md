# Flake Standardization — Requirements

**Date:** 2026-05-14
**Status:** Draft
**Scope:** Deep — cross-cutting infrastructure, documentation, and module export overhaul

## Problem Statement

This NixOS flake has strong infrastructure (43+ checks, 14 pre-commit hooks, full RFC 166 formatting stack) but its documentation is bloated (18 files, 380KB, 63KB README), modules lack external consumer essentials (no LICENSE, CONTRIBUTING, homeModules exports), and there is no CI pipeline. The goal is to bring documentation, module exports, and project presentation to the standard set by published module flakes (srvos, disko, catppuccin/nix) while preserving the existing quality infrastructure.

## Success Criteria

1. Documentation total drops from 380KB/18 files to ~100KB/~8 files (committed)
2. README is under 120 lines with clear module import instructions
3. External consumers can import any NixOS or HM module via standard flake patterns
4. mdBook documentation site builds via `nix build .#docs` with auto-generated option reference
5. Redundant module names fixed (e.g., `boot-boot`); existing taxonomy validated as philosophy-aligned
6. `nix flake check` runs in GitHub Actions CI on every push/PR
7. LICENSE and CONTRIBUTING.md exist at repo root
8. All auto-generated content is a build artifact, never committed

## Workstreams

### WS1: Documentation Restructure

**Goal:** Eliminate bloat while improving discoverability for both humans and AI.

**Principle:** If it can be derived from the code (via `nix eval`, `nix flake show`, or reading source), it should not be committed as a static file. Auto-generated content belongs in build outputs.

#### Keep (high-value, hand-maintained)

- `docs/STYLE.md` — code standards, hook-enforced, authoritative
- `docs/ARCHITECTURE.md` — directory layout, module boundary rationale (trim prose)
- `docs/INSTALLATION.md` — hard-won BTRFS+LUKS+disko knowledge
- `docs/SECURE-BOOT.md` — Lanzaboote operational reference
- `docs/SECRETS.md` — agenix operational reference
- `docs/NETWORKING.md` — DNS/Portmaster/Mullvad interaction reference

#### Merge

- `docs/BUILD.md` + `docs/TESTING.md` → `docs/DEVELOPMENT.md` (BUILD already covers testing)

#### Add

- `docs/USAGE.md` — external consumer getting-started guide (import instructions, example flake, available modules)
- `CONTRIBUTING.md` at repo root (not in docs/) — nixos-hardware style
- `LICENSE` at repo root — MIT

#### Remove (archive in git history, not in repo)

- `docs/DOC-STYLE.md` — meta-doc about docs, value doesn't justify carrying cost
- `docs/TERMINAL-TOOLS-INVENTORY.md` — personal tool catalog, not architecture
- `docs/STORAGE-STRATEGY.md` — decision record, belongs in commit history
- `docs/MIGRATE-MBP-SDB.md` — one-time migration guide, done
- `docs/REPO-DOC-TEMPLATE.md` — template for repos/, should live in repos/ if needed
- `docs/REPO-STANDARD.md` — move to repos/ or fold into CONTRIBUTING.md

#### Move to Build Artifacts (stop committing)

- `docs/OPTIONS.md` — generated by `nix build .#docs`, not committed
- `docs/options.json` — generated artifact
- `docs/host-template.nix.example` — generated artifact
- `docs/hm-host-template.nix.example` — generated artifact

#### README Rewrite

- Current: 63KB with auto-spliced sections
- Target: ~100-120 lines, hand-maintained
- Structure: badges, one-paragraph description, quick import example, host table, commands table, link to docs site, link to CONTRIBUTING

#### Remove Auto-Splice Hooks

- The `update-docs` hook currently splices generated content into README.md
- Rework: hook generates build artifacts only, README is static

### WS2: mdBook Documentation Site

**Goal:** Browsable documentation site with auto-generated option reference, built as a Nix derivation.

**Pattern:** Same as Nix's own manual — mdBook rendered from markdown sources.

#### Structure

```
docs/
  book.toml           # mdBook config
  src/
    SUMMARY.md         # Table of contents
    getting-started.md # → from USAGE.md
    architecture.md    # → from ARCHITECTURE.md
    development.md     # → from merged BUILD+TESTING
    style-guide.md     # → from STYLE.md
    installation.md    # → from INSTALLATION.md
    secure-boot.md     # → from SECURE-BOOT.md
    secrets.md         # → from SECRETS.md
    networking.md      # → from NETWORKING.md
    reference/
      options.md       # Auto-generated from nix eval (at build time)
      modules.md       # Auto-generated module catalog with descriptions
```

#### Build Integration

- `nix build .#docs` produces the mdBook site
- `nix run .#docs-serve` opens local preview
- GitHub Pages deployment via CI (optional, can be added later)

#### Auto-Generated Content

- Option reference: generated at build time from `nix eval`, injected into mdBook source
- Module catalog: generated from module metadata (docstrings + option counts)
- These are build-time steps, never committed to repo

### WS3: Module Name Cleanup (Minimal)

**Goal:** Fix only genuinely redundant or confusing names. Preserve the existing mechanism-first taxonomy.

**Principle:** The existing taxonomy already follows STYLE.md §13a's mechanism-first ladder. Categories like `macbook/`, `boot/`, `input/`, `sensors/` are CORRECT — they name concrete mechanisms, not abstract purposes. srvos-style categories (`mixins-*`, `roles-*`, `platform-*`) would introduce abstraction, violating the "named concretely" and "mechanism wins over intent" principles.

#### What Stays (and Why)

- `macbook/` — explicit vendor-category exception in §13a.1. Concrete. All 4 modules are Apple-specific patches.
- `boot/` — mechanism-first. Bootloader/kernel/initramfs IS the mechanism.
- `input/` — separate from `hardware/` because input devices have distinct driver/config concerns (libinput, firmware, HID).
- `sensors/` — separate from `hardware/` because sensor modules are monitoring/driver concerns (it87, nct6775, ryzen-smu).
- `diagnostics/` — turbostat IS a diagnostic tool, not hardware enablement.
- All other categories — already follow the taxonomy ladder.

#### Fix List (only genuinely redundant names)

- `boot-boot` → `boot-loader` or `boot-systemd-boot` — the only clearly redundant name (`parts/boot/boot.nix` exporting `boot-boot`)
- Audit remaining 75 modules for similar redundancies during planning (expect 0-2 more)

#### Migration (if any renames)

- Single atomic commit per rename
- Update all references (host configs, tests, docs, CLAUDE.md)
- `check-placement` hook validates automatically
- No deprecation aliases needed — no external consumers yet

### WS4: Module Export Completeness

**Goal:** Any module (NixOS or HM) consumable by external flakes via standard patterns.

#### NixOS Modules

- Add `nixosModules.default` alias (points to a meta-module that imports all, or omit if no sensible default — research showed nixos-hardware deliberately omits `.default`)
- Decision: likely NO `.default` — 76 modules, no sensible single import. Document this choice.

#### Home Manager Modules

- Export `homeModules.*` (not `homeManagerModules` — following catppuccin/nix convention for newer naming)
- Each HM module under `home/modules/` gets a corresponding `homeModules.<name>` export
- Add `_class = "home-manager"` annotation where missing

#### Library Exports

- Already correct (`flake.lib.*`). Verify all 6 helpers are documented in USAGE.md.

#### Consumer Integration Test

- Add a check that imports a single module into a minimal NixOS config and evaluates successfully
- Proves external consumption works, catches breaking changes

### WS5: CI/CD Pipeline

**Goal:** Automated quality gates on every push/PR.

#### GitHub Actions Workflow

```yaml
# .github/workflows/check.yml
name: Check
on: [push, pull_request]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: cachix/install-nix-action@v31
      - uses: cachix/cachix-action@v14
        with:
          name: <cache-name>
      - run: nix flake check -L
```

#### CI Checks

- `nix flake check` (runs all 43+ existing checks)
- Optional: DeterminateSystems/flake-checker for input health
- Optional: deploy mdBook site to GitHub Pages on main push

### WS6: Cleanup — update-docs Hook Rework

**Goal:** Stop auto-committing generated content. Generated docs become build artifacts.

#### Current Behavior (to change)

- `update-docs` hook in `parts/_build/git-hooks.nix` generates OPTIONS.md, options.json, templates
- Splices sections into README.md
- Re-stages generated files

#### New Behavior

- Hook removed or reduced to validation-only (warn if docs source changed without rebuilding)
- `parts/_build/docs.nix` (new) builds mdBook site + option reference + module catalog
- `nix build .#docs` is the only way to generate docs
- README.md is hand-maintained, no splicing

## Scope Boundaries

### In Scope

- All 6 workstreams above
- Updating CLAUDE.md/AGENTS.md to reflect new structure
- Minor module name fixes (boot-boot only, unless audit finds more)

### Deferred

- Per-module documentation files (76 is too many to maintain)
- GitHub Pages deployment (can add after mdBook works locally)
- Hydra/garnix continuous builds
- CODE_OF_CONDUCT.md (personal config, not community project)

### Out of Scope

- New tests or hooks beyond what's needed for the rename
- Changing the test framework or hook architecture
- Changing treefmt or formatter configuration
- Changing the dendritic architecture philosophy

## Dependencies and Assumptions

- **Assumption:** GitHub Actions runners support Nix via cachix/install-nix-action
- **Assumption:** mdBook is available in nixpkgs (it is: `pkgs.mdbook`)
- **Assumption:** boot-boot rename can be atomic (single commit) — no external consumers yet
- **Dependency:** WS3 (name cleanup) is small and fast — do first to avoid documenting wrong names
- **Dependency:** WS1 (doc restructure) and WS6 (hook rework) are coupled — do together
- **Dependency:** WS4 (exports) can happen independently
- **Dependency:** WS5 (CI) can happen independently after WS3

## Execution Order

1. **WS3** — Module name cleanup (small — fix `boot-boot`, audit others. Fast, do first.)
2. **WS1 + WS6** — Doc restructure + hook rework (coupled, biggest workstream)
3. **WS4** — Module export completeness (homeModules, integration test)
4. **WS2** — mdBook site (needs finalized docs)
5. **WS5** — CI pipeline (last, validates everything works)

## Risks

- **mdBook maintenance:** Another tool to keep updated. Mitigation: build from existing markdown, auto-generate reference content.
- **Hook rework regression:** Removing update-docs hook could break the doc freshness guarantee. Mitigation: replace with build-time generation that CI validates.
- **boot-boot rename:** Touches host configs, tests, docs. Mitigation: exhaustiveness hooks catch missing refs, `nix flake check --no-build` validates.
